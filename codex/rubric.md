# DSC 180B – Capstone Project Website Grading Checklist

## Baseline Requirements

**These are ALL must-haves.**

- [ ] **Public URL works (no login required)** ✅
- [ ] **Loads without errors (no broken build / blank page)** ✅
- [ ] **Mobile usable (readable and navigable on a phone)** ✅
- [ ] **No obvious placeholder content** (e.g., “lorem ipsum”, template text) ✅

**If any Must-Have is missing:** grade is capped at **B** (or **C** if multiple are missing). :contentReference[oaicite:0]{index=0}

---

# Clarity

_What is this project and why should I care?_

- [ ] **One-screen elevator pitch** (goal + why it matters) ✅
- [ ] Clear problem statement and target user/stakeholder
- [ ] Clear scope boundaries (what you did / didn’t do)
- [ ] Approach summary understandable to a technical reader outside the team
- [ ] Key results stated explicitly (not implied)
- [ ] Claims supported by evidence (plots, tables, examples, demos)
- [ ] Limitations / failure modes acknowledged (honest + specific)
- [ ] Guidance on how to interpret results (what’s meaningful vs noise)

**Boxes checked:** \_\_\_ / 8

**Grade rubric**

- **A:** ≥7 + all must-haves
- **B:** 5–6, ≤1 must-have missing
- **C:** ≤4 or ≥2 must-haves missing

---

# Simplicity & Information Architecture

**Goal:** A reviewer can reach the core story (**Problem, Method, Results**) in **two clicks**.

- [ ] **Navigation is obvious and consistent across pages** ✅
- [ ] **Core content reachable in ≤2 clicks from home (Problem, Method, Results)** ✅
- [ ] Shallow structure (no maze of nested pages/menus)
- [ ] Page titles match what the content actually contains
- [ ] Low cognitive load: scannable headings, short paragraphs, good whitespace
- [ ] No duplicated or conflicting sections (e.g., two “Results” pages disagree)
- [ ] Links behave correctly (no dead links / wrong anchors / ambiguous “click here”)
- [ ] Important assets embedded or clearly linked (report, repo, poster, demo)

**Boxes checked:** \_\_\_ / 8

**Grade rubric**

- **A:** ≥7 + all must-haves
- **B:** 5–6, ≤1 must-have missing
- **C:** ≤4 or ≥2 must-haves missing

---

# Effectiveness

_Does the site showcase the work convincingly?_

- [ ] **Results are front-and-center (not buried below long methodology)** ✅
- [ ] Strong visuals that carry the story (not decorative)
- [ ] Evaluation is explicit (metrics, baselines, comparisons, ablations as relevant)
- [ ] Rigor: dataset description + splits + leakage/validity considerations
- [ ] Shows iteration: what changed after early attempts and why
- [ ] Includes “so what?”: implications, deployment considerations, next steps
- [ ] Clear contributions: what the team built vs reused
- [ ] Suitable for an external audience (an industry mentor can judge quickly)

**Boxes checked:** \_\_\_ / 8

**Grade rubric**

- **A:** ≥7 + all must-haves
- **B:** 5–6, ≤1 must-have missing
- **C:** ≤4 or ≥2 must-haves missing

# Communication Quality

_Readable, credible, appropriately technical._

- [ ] **Writing is concise and free of major grammar/typos** ✅
- [ ] Terms defined once; acronyms explained
- [ ] Figures have captions/labels; axes readable; metrics defined
- [ ] Avoids hype unless supported by evidence
- [ ] Citations/credits for external datasets/tools/models (where relevant)
- [ ] Tone matches audience: technical but not a wall of text

**Grade rubric**

- **A:** ≥5 + all must-haves
- **B:** 3–4, ≤1 must-have missing
- **C:** ≤2 or ≥2 must-haves missing

# Technical & Trust Signals

_Lightweight but important._

- [ ] **Links to repo and reproducibility artifacts are present** ✅
- [ ] Demo instructions clear (if demo exists) **OR** demo video present
- [ ] **Sensitive data handled appropriately** (no secrets, keys, private data leaks) ✅
- [ ] Page load performance acceptable (no massive uncompressed assets)
- [ ] Accessibility basics: contrast/readability; alt text for key images (best effort)
